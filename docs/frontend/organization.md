# Frontend Architecture

## Overview

KF2 has two browser SPAs served from a single Rust server:

- **Projector** — renders video, lyrics, queue display, and pitch visualization on a big screen
- **Remocon** — phone-sized remote control for song search, queue management, and playback controls

Both SPAs live in `src/frontend/` as a pnpm workspace monorepo, sharing code through internal packages.

## Workspace Structure

```
src/frontend/
├── pnpm-workspace.yaml       # declares packages/*
├── package.json              # root scripts: generate, dev, build
├── tsconfig.base.json        # shared TypeScript compiler options
├── buf.yaml / buf.gen.yaml   # protobuf codegen config
│
└── packages/
    ├── proto/                # generated TS protobuf code (@kf2/proto)
    ├── common/               # shared library (@kf2/common)
    ├── projector/            # projector SPA (@kf2/projector)
    └── remocon/              # remocon SPA (@kf2/remocon)
```

### `@kf2/proto`

Contains TypeScript message types and service descriptors generated by `buf generate` from `proto/*.proto`. The `gen/`
directory is gitignored — run `just generate` (or `pnpm run generate` from `src/frontend/`) to regenerate after proto
changes.

### `@kf2/common`

Shared library consumed by both SPAs. Organized by concern:

- **`api/`** — Connect-ES gRPC-Web transport and a custom RTK Query `baseQuery` that bridges Connect-ES clients to Redux
- **`store/`** — RTK Query API slices (e.g. `sessionApi`) and shared Redux slices
- **`hooks/`** — React hooks wrapping store selectors and RTK Query endpoints (e.g. `usePlaybackState`, `useQueue`)
- **`components/`** — shared UI components (e.g. `Loader`)

### `@kf2/projector` and `@kf2/remocon`

Each SPA follows the same internal pattern:

- **`features/<name>/`** — self-contained feature modules (components, CSS modules, feature-local slices)
- **`pages/`** (remocon only) — route-level components composing feature components
- **`store.ts`** — `configureStore` combining shared slices from `@kf2/common` with app-specific slices
- **`App.tsx`** — router and top-level layout
- **`main.tsx`** — React root with Redux `Provider`

## Feature Folder Convention

Each `features/<name>/` directory is self-contained:

```
features/playback/
├── PlaybackControls.tsx
├── PlaybackControls.module.css
├── PitchControls.tsx
├── PitchControls.module.css
├── playbackSlice.ts          # feature-local state (if needed)
└── index.ts                  # barrel export
```

Pages in `remocon/pages/` compose feature components into route-level views.

## Data Flow

```
proto/*.proto
    │ buf generate
    ▼
@kf2/proto (TS messages + service descriptors)
    │ imported by
    ▼
@kf2/common/api (Connect-ES transport + RTK Query baseQuery)
    │ used by
    ▼
@kf2/common/store (RTK Query API slices — sessionApi, etc.)
    │ hooks auto-generated
    ▼
React components (useListSessionsQuery, useCreateSessionMutation, etc.)
```

**Unary RPCs** — RTK Query API slices define endpoints. The custom `baseQuery` wraps Connect-ES's `createClient` to make
typed gRPC-Web calls. Components use auto-generated hooks.

**Server streaming** (future) — RTK Query's `onCacheEntryAdded` lifecycle will open gRPC server streams via Connect-ES,
dispatching updates into the cache as they arrive.

**Local UI state** — plain `createSlice` in Redux Toolkit, no API involvement.
